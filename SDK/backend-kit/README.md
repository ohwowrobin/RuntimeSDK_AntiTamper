# Korvayne Runtime Developer Endpoint Kit

This kit is optional. It helps a studio self-host its own telemetry and access-check endpoints for
Korvayne Runtime if it wants server-side correlation. The SDK runs fully without it; no Korvayne
server is involved, and no player telemetry leaves the studio's own infrastructure.

## Contents

```text
schemas/telemetry-event.schema.json          JSON Schema for detection events
schemas/access-check-request.schema.json     JSON Schema for SDK access-check requests
schemas/access-check-response.schema.json    JSON Schema for access decisions

cloudflare-worker/                           Cloudflare Worker example endpoint
```

## Security Model

The endpoint URL is public once the game ships. Do not rely on hiding the URL and do not put a
server secret in `anticheat.ini`.

Use this flow instead:

1. Player logs in through the studio/game backend.
2. Backend verifies platform identity, session state, and game build.
3. Backend issues a short-lived signed token for that player/session/build.
4. Game passes the token to the SDK with `AC_SetTelemetryToken(...)`.
5. Telemetry/access endpoints validate the token and bind request fields to the token claims.
6. Endpoint rate-limits by token, player, session, game ID, and build.
7. Endpoint rejects duplicate `event_id` values to block replay spam.
8. Game server still enforces bans before matchmaking or server join.

Telemetry and access requests may include studio-supplied correlation fields such as `build_channel`
or `deploy_id`. Treat them as useful correlation fields, not as authentication secrets.

Telemetry is still client-originated data. The endpoint templates therefore stamp accepted records as
`client_reported` unless the session token contains `runtime_trust: "sdk_attested"` from a studio
attestation/liveness flow. Non-attested records should feed review/correlation, not automatic bans.
If your integration has a trusted runtime proof, enable `KORVAYNE_REQUIRE_TRUSTED_RUNTIME=1` to fail
closed when that proof is absent. If your access policy needs a live SDK heartbeat, also enable
`KORVAYNE_REQUIRE_RECENT_TELEMETRY=1`.

Detection event categories are defined in `schemas/telemetry-event.schema.json`. Korvayne Runtime currently
uses `injection`, `hook_detection`, `handle_checks`, `debugger`, `boot_state`, `memory_integrity`,
`sdk_integrity`, `protected_value`, `access_check`, `aim_behavior`, and `enforcement`. The
`unknown` category is a forward-compatibility fallback for future/unmapped sensors; store it for
review, but do not treat it as high-confidence evidence by itself.

## Token Contract

`runtime_session_token` is not generated by Korvayne. It is a normal runtime credential from the
studio backend to the game client. The usual flow is:

1. The player signs in through Steam, Epic, a launcher account, or the studio backend.
2. The trusted backend verifies that login and creates a short-lived token for this exact
   `game_id`, `player_id`, `session_id`, and `game_build`.
3. The game receives that token with the normal login/session response.
4. The game calls `AC_SetTelemetryToken(token)` before telemetry or access checks are sent.
5. The telemetry/access endpoint validates the token signature or lookup result and rejects requests
   where the JSON fields do not match the token claims.

The example templates use a compact HMAC token:

```text
base64url(json-payload).base64url(hmac_sha256(payload, server_secret))
```

Example payload:

```json
{
  "jti": "1bb86a62-3b34-4ae0-a0e6-9e1f9c53426e",
  "scope": ["telemetry", "access"],
  "game_id": "example-game",
  "player_id": "player_123",
  "session_id": "match_abc",
  "game_build": "1.0.42",
  "runtime_trust": "client_reported",
  "iat": 1782753720,
  "exp": 1782754620
}
```

Studios can replace this with JWT, PASETO, opaque session IDs, or their existing auth system. The
requirements stay the same: short TTL, server-side signature/lookup, and binding to player/session.

## SDK Config

```ini
[Telemetry]
enabled = 1
endpoint = https://your-game.example.com/anti-cheat/events
token_source = runtime_session_token

[AccessCheck]
enabled = 1
endpoint = https://your-game.example.com/anti-cheat/access-check
mode = startup_and_recheck
```

Telemetry and AccessCheck use the same configured auth header and runtime token. The game should set
that token with `AC_SetTelemetryToken(...)` after the studio backend has verified the player/session.
Do not ship the backend signing secret or a long-lived server credential in `auth_token`.

Example login/session response from the studio backend:

```json
{
  "player_id": "player_123",
  "session_id": "match_abc",
  "korvayne_token": "short-lived-token-created-by-your-server"
}
```

Game code:

```c
AC_SetGameId("example-game");
AC_SetPlayerId(player_id);
AC_SetSessionId(session_id);
AC_SetGameBuild("1.0.42");
AC_SetTelemetryToken(korvayne_token);
```

## Production Checklist

- Use HTTPS only.
- Keep signing secrets only on trusted backend infrastructure.
- Reject missing or mismatched `game_id`, `player_id`, `session_id`, and `game_build`.
- Record studio-supplied correlation fields (e.g. `build_channel`, `deploy_id`) when present, but do not trust them instead of the session token.
- Reject expired tokens and unknown session IDs.
- Rate-limit before expensive processing.
- Deduplicate `event_id` for at least 24 hours.
- Derive IP address server-side from the HTTP request; do not accept client-provided IP fields.
- Store only the fields the studio needs, with a defined retention window.
- Never auto-ban from one client event without server-side correlation.

## Picking a Template

- Use `node-express/` if the studio already runs a Node backend.
- Use `cloudflare-worker/` if the studio wants a cheap edge endpoint in front of its existing backend.

Both examples are intentionally small. Replace in-memory maps with a shared store before production.

## Optional Signature Update Channel

Korvayne Runtime can pull signed detection-signature updates from a self-hostable update channel. This
is optional and advanced: the SDK ships with a working built-in signature set and arms fully without
it. Host the channel yourself if you want to push new detection signatures to deployed clients between
releases.

## Scope

Korvayne Runtime is a user-mode client-integrity SDK. It is not a kernel anti-cheat and not a
standalone ban oracle. Treat its detections as signals to correlate server-side, not as automatic
verdicts. Never auto-ban from a single client event without server-side correlation.
